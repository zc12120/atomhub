# AtomHub 前端中文化设计

## 目标
将 AtomHub 前端网页中所有用户可见文案改为中文，同时保留 `token` 这一单词不翻译。

## 范围
本次只修改前端可见文本，包括：

- 登录页
- 顶部栏
- 左侧导航
- Dashboard 页面
- Keys 页面
- Models 页面
- Requests 页面
- Health 页面
- 加载态、空状态、错误提示、按钮、表头、筛选标签等页面文案
- 与页面文案直接相关的前端测试断言

## 不在范围内
以下内容保持不变：

- URL 路径，如 `/login`、`/dashboard`、`/requests`
- 后端接口路径
- API 字段名与 JSON key
- 数据库结构
- 业务逻辑
- README 与后端返回报错文案
- `token` 一词本身

## 约束
- 只做前端中文化，不引入 i18n 框架。
- 文案直接保存在现有页面/组件文件中，不额外抽象文案系统。
- 已有测试中如果断言了英文文案，需要同步改成中文断言。

## 实现方式
### 方案选择
采用“直接替换页面可见文案”的方式。

### 原因
- 改动最小，最适合当前自用项目。
- 不增加额外架构复杂度。
- 能最快完成你要的前端体验调整。

## 具体修改点
### 登录页
将以下内容中文化：
- 标题
- 说明文字
- 用户名/密码标签
- 登录按钮
- 登录中提示
- 登录失败相关提示

### 全局框架
将以下内容中文化：
- 顶部标题与副标题
- “已登录为某用户”一类提示
- 登出按钮
- 左侧导航名称
- 会话检查中的加载提示

### Dashboard 页面
将以下内容中文化：
- 页面标题
- 加载提示
- 错误提示
- 汇总卡片标题
- 表格表头

说明：涉及 `token` 的标题保留 `token` 单词，例如：
- Prompt Tokens -> Prompt token
- Completion Tokens -> Completion token
- Total Tokens -> Total token

### Keys 页面
将以下内容中文化：
- 页面标题
- 新增 key 表单标题
- 所有表单标签
- 保存/探测/编辑/启用/禁用/删除/取消等按钮
- 表格表头
- 加载提示、错误提示、空状态提示
- 行内编辑相关标签

说明：`key` 是否翻译为“密钥”按页面语义统一处理；但 `token` 不翻译。

### Models 页面
将以下内容中文化：
- 页面标题
- 加载提示、错误提示
- 表格表头
- 空状态提示

### Requests 页面
将以下内容中文化：
- 页面标题
- 模型筛选标签
- 汇总卡片标题
- 模型统计表表头
- 请求记录表表头
- 加载提示、错误提示、空状态提示

说明：所有 token 相关列名继续保留 `token` 单词。

### Health 页面
将以下内容中文化：
- 页面标题
- 加载提示、错误提示
- 汇总卡片标题
- 表格表头
- 空状态提示

## 文案规范
- 中文优先，自然、简洁，不追求逐字翻译。
- `token` 不翻译。
- 术语尽量统一：
  - Dashboard -> 仪表盘
  - Requests -> 请求记录
  - Health -> 健康状态
  - Models -> 模型
  - Keys -> 密钥
- 若某个技术词翻译后会更别扭，可保留英文，但本次默认尽量中文化。

## 测试策略
- 更新前端页面测试中所有受影响的可见文案断言。
- 运行前端测试与构建，确保：
  - 文案修改未破坏现有页面渲染
  - 路由与功能行为不受影响

## 风险与处理
### 风险 1：测试断言失效
处理方式：同步更新测试中的可见文字断言。

### 风险 2：同一文案在多个页面不一致
处理方式：按页面批量检查，统一术语。

### 风险 3：`token` 被误翻译
处理方式：最终人工回看所有 token 相关标题与列名。

## 验收标准
满足以下条件即可认为完成：

1. 前端网页中用户能看到的英文文案都已改为中文。
2. `token` 一词仍保留英文。
3. URL 路径和 API 字段名未改动。
4. 前端测试通过。
5. 前端构建通过。
